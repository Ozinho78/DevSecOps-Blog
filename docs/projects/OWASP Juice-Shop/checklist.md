# Project Submission Checklist – Juice Shop Hacking

Please complete all items on this list before submitting your project.
If you have added any extras, mention them briefly so the mentors can review them if needed.

---

## 1. Challenges

### Choice of Challenges

- [x] Select a minimum of 2 and a maximum of 4 hacking challenges within the Juice Shop that you want to solve and document independently
- [x] Your challenges must not all come from the same category — for example, three videos with XSS attacks are not permitted

### Submission Video

- [x] For each selected challenge, a video must be prepared, recorded, and added as part of the documentation
- [x] Each video must not exceed 5 minutes
- [x] Your video should include the documentation of the challenge as well as the execution of the cyberattack, while you comment on and briefly explain the steps taken

### Solutions and Materials

- [x] The main README must contain a Table of Contents (ToC)

---

## 2. Documentation

- [x] Documentation of the code and the project must be provided in the repository in the form of a Markdown file
- [x] The documentation language for all projects (and related materials) is English

### Project Structure

- [x] The existing Docusaurus project is used
- [x] There is a main folder named **"Juice Shop Master"**
  - [x] It contains a `README.md` with the following elements:
    - [x] Exactly one `<h1>` at the very top of the README
    - [x] Immediately below: a project description (max. 2–5 sentences)
    - [x] A Table of Contents (ToC)
    - [x] A Quickstart section
- [x] For each challenge there is a dedicated subfolder
  - [x] It contains the challenge documentation
  - [x] It contains images where applicable
- [x] The challenge documentation is linked in the main README

### General

- [x] The README includes a notice that the content is for educational purposes only
- [x] Each challenge solution and the resources used are presented in a video of no more than 5 minutes
- [x] All challenge videos are linked in the main README of the submission repository
- [x] For each challenge, the category is stated and a brief explanation is provided of the dangers of the exploited vulnerability and the potential consequences of being susceptible to this type of security flaw

### Security Notes

- [x] No real personal data or data related to yourself is used during hacking
- [x] No SSH keys are stored in the Git repository workspace
- [x] No passwords, tokens, or usernames are stored in the code — environment variables are used instead
- [x] No IP addresses or other sensitive information are stored in the Git repository

---

## 3. Testing

Before submitting your project, ensure and verify the following:

- [x] Each challenge is briefly explained, including how the vulnerability is exploited and what steps are required
- [x] The hacks can be reconstructed and reproduced using the video, and work as demonstrated
